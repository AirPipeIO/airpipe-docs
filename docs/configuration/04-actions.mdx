# Actions

Actions are the steps an interface runs when it is called. Each action does one
thing — fetch data or transform it — and actions can be ordered, retried, and
run conditionally on each other's results.

## What an action can do

- **Input** — where the action's data comes from: [`input`](/docs/configuration/interfaces/actions/inputs)
  to select request data or a previous action's output, a fetch
  ([`http`](/docs/configuration/interfaces/actions/inputs/http),
  [`database`](/docs/configuration/interfaces/actions/inputs/databases),
  [`command`](/docs/configuration/interfaces/actions/inputs/command),
  [`email`](/docs/configuration/interfaces/actions/inputs/email)), or a
  [`lookup`](/docs/configuration/interfaces/actions/lookups) to fan out over an array.
- **Validate** — [`assert`](/docs/configuration/interfaces/actions/asserts) tests against the data.
- **Transform** — [`post_transforms`](/docs/configuration/interfaces/actions/transforms/functions) to reshape it.
- **Order & flow** — `run_when_succeeded`, `run_when_failed`, `depends_on`
  (see [Workflow control](/docs/configuration/interfaces/actions/workflow-control)),
  plus `retry`, `timeout` and `delay`.
- **Response shaping** — `hide_data_on_success`, `output`, `response_on_error`, and more.

Data flows between actions with interpolation — reference a previous action's
output as `a|ActionName::field|`. See [Interpolation](/docs/configuration/interpolation).

```yaml
actions:
  - name: FetchUser
    http:
      url: https://api.example.com/users/a|params::id|
  - name: Notify
    run_when_succeeded: [FetchUser]
    http:
      url: https://hooks.example.com/notify
      body: { email: a|FetchUser::email| }
```

For the **complete field list** with every option, type and default, see the
**[Actions reference](/docs/reference/actions)** — generated from the engine
schema, so it is always current.
