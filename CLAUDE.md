## What this is

Docusaurus 3 documentation site for **Air Pipe**, published to https://docs.airpipe.io. Content is MDX + YAML config examples; there is almost no application code. Node `>=18`.

## Commands

- `npm start` — dev server with live reload (runs `gen` first via `prestart`)
- `npm run build` — production static build into `build/` (runs `gen` first via `prebuild`)
- `npm run serve` — serve the built `build/` locally
- `npm run clear` — clear the `.docusaurus` cache (run when the dev server shows stale content or errors after moving/renaming files)
- `npm run gen` — regenerate the auto-generated docs from committed snapshots (no network)
- `npm run gen:refresh` — re-fetch the live APIs and rewrite `data/*.snapshot.json` (commit the result)

No test suite, no linter, no formatter are configured. Verification = build succeeds and pages render. `onBrokenLinks` is set to `throw`, so a broken internal link fails the build.

## Generation harness (auto-populated docs)

Large parts of the docs are **generated from live Air Pipe sources**, not hand-written:

- **Configuration reference** ← `https://api.airpipe.io/config/schema` (schemars JSON Schema of `IntegrationConfig`). Generator: `scripts/gen-config-reference.mjs` → `docs/reference/*.md`. Hand-authored concept + example prose lives in `reference-content/<page>.md` (committed) and is **injected** into the matching generated page after its H1, before the field tables — so each type has ONE page combining stable narrative with always-current schema tables.
- There is **no separate Configuration tab**. The **Reference** navbar tab is a single sidebar with two groups (`sidebars.js` `referenceSidebar`): **Guides** (`docs/reference-guides/*`, committed hand-authored concept pages — config structure, globals, interpolation, substitutions, managed variables & secrets, workflow control, lookups, CLI) pinned above **Schema reference** (`docs/reference/*`, generated). Guide pages keep their original `/configuration/*` slugs (location-independent), so inbound links resolve unchanged.
- Both generators run their output through `closeMarkers()` (`scripts/lib/sources.mjs`) to normalize deprecated open-form interpolation (`a|body` → `a|body|`) coming from schema doc-comments / pack content.
- **Marketplace examples** ← `https://api.airpipe.io/accounts/marketplace/public/packs` + `/public/pack?slug=`. Generator: `scripts/gen-marketplace.mjs` → `docs/examples/marketplace/<category>/*.md`.

Two-stage design (`scripts/lib/sources.mjs`):
1. `npm run gen:refresh` hits the live APIs and writes `data/schema.snapshot.json` + `data/packs.snapshot.json`. **These snapshots are committed** — they are the offline source of truth.
2. `npm run gen` (wired into `prestart`/`prebuild`) reads only the snapshots and emits Markdown. No network at build time → deterministic, offline, CI-safe. Live changes reach the docs when someone runs `gen:refresh` and commits.

Conventions that matter for the generated content:
- Generated pages are **`.md`, not `.mdx`**, and `docusaurus.config.js` sets `markdown.format: "detect"` so `.md` parses as lenient CommonMark. Pack READMEs and schema doc-comments contain `{` and `<` (YAML examples) that would break the strict MDX parser.
- Generated output lives in **routed dirs** (`configuration/reference/`, `examples/marketplace/`) — NOT `_`-prefixed, or Docusaurus would exclude them from routing. They are gitignored (`.gitignore`); the snapshots are not.
- The API response envelope mirrors Air Pipe's action model: payloads live at `data.List.data` (packs list), `data.GetPack.data` (pack detail), `data.<Action>.data` generally. `readme_content` is usually null — the README is the `README.md` file inside the pack's `files[]` (base64 `content`).
- Config-reference cross-type links use explicit heading ids (`## Type {#type-<slug>}`) to avoid github-slugger collisions with page H1 titles, and relative `./<file>.md#...` links so they resolve regardless of `routeBasePath` (`/docs`).

## Content structure

All docs live under `docs/`. Four independent sidebars, each **autogenerated** from one top-level directory (see `sidebars.js`) and surfaced as a navbar tab (`docusaurus.config.js`):

| Directory | Sidebar / navbar tab |
|---|---|
| `docs/documentation/` | Docs |
| `docs/configuration/` | Configuration |
| `docs/tutorials/` | Tutorials |
| `docs/examples/` | Examples |

Because sidebars are autogenerated, adding a `.mdx` file to one of these directories automatically adds it to the nav — there is no manual sidebar list to edit.

### Conventions that matter

- **Numeric prefixes control order, not URLs.** `01-`, `02-`, `05-inputs/` etc. set sidebar ordering and are stripped from the generated URL. Renumber siblings to reorder.
- **`_`-prefixed files and directories are partials** — Docusaurus excludes them from routing. They are the reuse mechanism: authored once, then `import`ed into a page as an MDX component. Example: `docs/configuration/_transform_functions/_filter.mdx` is imported into `docs/configuration/07-transforms.mdx`. To add a transform doc, create the partial there and import it into the transforms page.
- **Cross-section content reuse** is also done by import: example pages pull whole tutorial pages in, e.g. `import Tutorial from "../../tutorials/Command to API/01-json-out-to-api.mdx"` then `<Tutorial/>`.
- **Custom `slug:` in frontmatter overrides the path-based URL** (e.g. `slug: /examples/parse-json-to-api`). Internal links use these slugs, so changing a slug breaks links elsewhere — grep before editing one.

### YAML example configs (important gotcha)

`.yml` files under `docs/examples/_ex_ymls/`, `docs/examples/Google/configs/`, etc. are the source-of-truth Air Pipe configs. They are embedded into pages via the `docusaurus-theme-github-codeblock` theme using a `reference` fence pointing at a **GitHub URL on the `main` branch**:

```
```yml reference title="List Assistants"
https://github.com/airpipeio/airpipe-docs/blob/main/docs/examples/_ex_ymls/_openai/list-assistants.yml#L1-L18
```
```

Consequences:
- Edits to a referenced `.yml` **only render after the change is merged to `main`** — local dev shows the old committed version, not your working copy.
- Line ranges (`#L1-L18`) are hardcoded; adding/removing lines in the `.yml` shifts them and silently changes what renders.
- Example pages also have a "Deploy" button linking to `app.airpipe.io/configuration/load?yml=<raw main-branch URL>`.

## Site-wide behavior

- **Dark mode only** — `colorMode.disableSwitch: true` in `docusaurus.config.js`; no light theme is styled.
- Mermaid diagrams enabled (` ```mermaid `), local search (`@easyops-cn/docusaurus-search-local`), GitHub-linked code blocks.
- Global custom scripts injected via config `scripts`: `static/js/sidebar-toggle.js` and `static/js/strip-toggle.js`.
- Reusable MDX React components live in `src/components/mdx-comps/` (`Cards`, `Paper`); homepage/landing in `src/pages/` and `src/components/HomepageFeatures/`.

## Deployment

Production is served at the site root (`baseUrl: "/"`, `url: https://docs.airpipe.io`). The GitHub Actions deploy workflow is currently **disabled** (`.github/workflows/deploysite.yml.disable`).
